重构测试方案

确定重构范围

  1. 确定测试计划 - 测试时间， 业务量

  2. 确定重构影响范围

    1. 前端无改动，仅后端语言和部分接口重构
    2. 接口方面，改动包含仅读，还是读写；
    3. 写的话对应其写数据（DB/redis）的变更， 考虑调用该DB的其他业务是否会用影响；
  1. 测试方案确定，确定风险规避：

    1. 接口的变更对前端的影响 -> 业务覆盖前端

    2. 从以下几点检查对调用变更接口的后端服务的影响：

      • DB config – 查看property文件
      • 根据数据变化 – 查看相关配置文件
      • DB 发MQ， MQ相关的消费者

  2. 测试本身

    1. Request

    2. Response - 读/写

    3. 数据层（Data） - 对应写数据

      ​

测试方案

针对4.1

Request验证： 线上拉取用户请求，在测试环境请求，主要查看返回error ， schema 信息，暂时不用关注data，也不需要去断言数据结果;

后期将API ， 请求各种类型的params 封装成一个set集合使用；

针对4.2

Response验证:

  1. 使用goreplay 工具录制遍历老版本的所有请求操作，捕捉log中的ip, request ,response 结果；
  2. 在新版本服务中，回放replay所有的请求，查看所有的结果是否一致，包括写data是否一致，即数据库数据；

针对4.3

  1. 使用老版本服务写入一套数据，查看数据库；
  2. 清除数据库（如果不在一个库内，不用清除），再使用新版本服务写入一套数据，查看数据库数据；
  3. Compare 两组数据
  4. 方法： a.可以人工对比； b. 写一组脚本，包含config文件，old 服务数据， new 服务数据，运行文件，结果文件； c. 优化，生成报告
